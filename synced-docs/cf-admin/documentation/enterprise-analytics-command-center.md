# Enterprise AI Command Center: Architecture & Implementation

## 1. Overview & Paradigm Shift

The Enterprise AI Command Center is a comprehensive analytics overhaul for the Madagascar Chatbot. The previous analytics implementation relied on mock data in the frontend and lacked the backend infrastructure to track meaningful business outcomes. It focused heavily on raw, system-level metrics (e.g., token counts, raw latency) rather than user satisfaction and resolution success.

We executed a paradigm shift from tracking **technical events** to tracking **customer outcomes**. The new architecture answers critical business questions:
- Is the AI actually containing and resolving issues, or just deflecting them?
- Are users satisfied with the AI's responses (CSAT)?
- What specific knowledge gaps exist, grouped by topic rather than a raw list of thousands of queries?
- What is the true API cost per successful resolution?

## 2. What We Built: The 3-Pillar Dashboard UI

The `AnalyticsDashboard.tsx` in the `cf-admin` project was completely rewritten. All fake/placeholder data was removed in strict compliance with project rules. The dashboard now utilizes a high-density, 3-Pillar layout connected to real, live data from Supabase.

### Tier 1: Executive ROI (Outcomes)
This section displays the ultimate value generated by the AI agent:
- **Containment Rate:** The percentage of conversations resolved by the AI without human escalation or user abandonment.
- **Global CSAT:** The average customer satisfaction score (e.g., 1-5 stars or thumbs up/down).
- **Cost / Resolution:** Total AI API cost divided by the number of successfully contained sessions.
- **Avg Handle Time:** The average duration of resolved sessions.

### Tier 2: AI Quality & Diagnostics (Operations)
This section provides operational diagnostics for prompt engineering and model evaluation:
- **Total Conversations & Rates:** Absolute volume, Fallback Rate (when the AI admits it doesn't know), Escalation Rate, and Abandonment Rate.
- **Model Distribution & Performance:** A visual breakdown of which models (e.g., Claude 3.5 Haiku, Llama 3) are handling traffic, comparing their share, average latency, and knowledge base hit rates.
- **Top Intents:** A bar chart showing the most common semantic intents (e.g., booking, pricing, amenities) derived from user queries.

### Tier 3: Actionable Intelligence (Clustered KB Gaps)
Instead of a raw table of failed queries, this section groups missing knowledge into actionable "Topics".
- Displays the Topic Cluster, total volume of failures, sample user queries, and the timestamp it was last seen.
- Enables administrators to quickly identify and prioritize the most impactful knowledge base additions.

## 3. How It Works: Backend Implementation Details

To power the dashboard without increasing AI token costs or overloading the database, we implemented a highly efficient, RPC-driven backend architecture.

### A. Supabase Schema Expansion (Migration 0004)
We expanded the database schema to capture business outcomes:
1.  **`feedback_events` Table:** A new table created to store CSAT scores and comments tied to specific conversations, contacts, and messages. Secured with Row Level Security (RLS).
2.  **`conversation_metrics` Extensions:** Added columns for `containment_status`, `csat_score`, `fallback_count`, `user_device`, and `user_country`.
3.  **`kb_gaps` Extensions:** Added a `cluster_topic` column to allow grouping of similar queries.
4.  **Updated `increment_conversation_metrics` RPC:** We modified the core telemetry tracking function to accept a `p_model` parameter and to increment the `fallback_count` whenever a KB miss occurs, enabling accurate model distribution and fallback tracking per conversation.

### B. The "Mega-RPC" Aggregation Pattern
To ensure the dashboard loads instantly without a waterfall of API requests, we built two powerful Postgres Remote Procedure Calls (RPCs):

1.  **`get_command_center_analytics(p_days)`:** 
    - This "Mega-RPC" aggregates data across `conversations`, `conversation_metrics`, `feedback_events`, and `intent_events` for a given time period (e.g., 7 days).
    - It performs complex SQL calculations (e.g., containment rate math, grouping models by usage, grouping top intents) entirely within the database layer.
    - It returns a single, perfectly structured JSONB payload containing the `executive`, `operations`, `models`, and `intents` data structures expected by the frontend.
2.  **`get_kb_clusters(p_resolved)`:**
    - This RPC aggregates `kb_gaps` by the `cluster_topic` (falling back to the raw `query_text` if unclustered).
    - It uses derived tables and `jsonb_agg` to return the total volume, the most recent timestamp, and an array of distinct sample queries for each cluster.

### C. Zero-Blocking Edge Telemetry Pipeline
In the `cf-chatbot` worker (`src/storage/supabase.ts`), telemetry data is collected using a strict **fire-and-forget** pattern. 
- When a user sends a message, the worker generates the response and immediately returns it to the user.
- *After* the response is sent, `Promise.allSettled` is used to asynchronously insert intent events, log KB gaps, and call the `increment_conversation_metrics` RPC. 
- We updated this pipeline to pass the `modelUsed` metadata to the RPC, ensuring the dashboard accurately reflects which LLM handled the turn.
- This guarantees that telemetry collection never adds latency to the chat experience or crashes the main pipeline.

### D. Cloudflare Worker API Routes
We exposed the new RPCs via the `cf-chatbot` Admin API (`src/admin/api.ts`):
- `GET /admin/analytics/command-center?days=N`
- `GET /admin/analytics/kb-clusters?resolved=false`

These routes are protected by the `X-Admin-Key` authorization header and seamlessly proxy data from Supabase to the `cf-admin` dashboard.

## 4. Cost & Resource Impact

A critical requirement of this overhaul was to avoid inflating operational costs.

- **Zero Additional AI Tokens:** The entire dashboard is populated via SQL aggregations on structured telemetry data. We do not use LLMs to summarize or generate the analytics reports.
- **Negligible Database Load:** 
  - Writes: Telemetry is written using highly optimized upserts per conversation turn. We simply added one parameter (`p_model`) to an existing RPC call; we did not add new write queries to the critical path. The `feedback_events` table only writes when a user actively submits a rating.
  - Reads: The dashboard triggers exactly two highly optimized RPC reads per page load.
- **Zero Additional Worker Costs:** The new endpoints are part of the existing `cf-chatbot` worker. Cloudflare bills per-request, and these analytics routes are only hit when an administrator views the dashboard.

## 5. Future Enhancements

The foundational architecture is now complete and live. Future planned phases include:
1.  **CSAT UI Widget:** Building the frontend widget in the chat interface to allow users to submit the thumbs up/down ratings that populate the `feedback_events` table.
2.  **Background AI Clustering Cron:** Implementing a scheduled worker (Cron Trigger) that periodically processes unclustered `kb_gaps` rows, using an LLM to assign semantic `cluster_topic` labels, thereby fully activating the Tier 3 Actionable Intelligence pillar.
